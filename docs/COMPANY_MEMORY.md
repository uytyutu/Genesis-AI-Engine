# Company Memory

**Status:** Foundational — institutional memory, not file storage.  
**Implementation:** future layer on top of Audit + analytics; **not built yet**.

**Related:** `COMPANY_OS.md` · `LIFECYCLE.md` · `FUTURE_LAYERS.md` (Evolution)

---

## What Company Memory is

**Not** “where we save files.”

**Company Memory** is Genesis learning as an organization:

> Last year we built 843 websites.  
> Restaurant sites sold best.  
> Template v7 increased sales 18%.  
> Users preferred green themes.

Memory compounds **decisions** — not just logs.

---

## What Memory is not

- Replacing the owner’s judgment
- Hidden training on client secrets without consent
- Guaranteed profit predictions
- Automatic changes without approval (Evolution proposes → owner approves)

---

## Data sources (future)

| Source | Feeds memory |
|--------|----------------|
| Audit log (`audit.jsonl`) | What happened, when, outcomes |
| Task / queue history | Throughput, failures |
| QA results | Quality patterns |
| Sales / Payment Hub | What sells, at what price |
| Template analytics | Version A vs B |
| Owner approvals / rejects | Taste and strategy |
| Ideas feedback | Which niches owner picks |

Kernel and Brain **write events**. A future **Memory Service** (Research / Evolution department) **reads and summarizes** with explainable “why.”

## Example insights (year 2)

```
Products
  Total built: 843
  Best vertical: restaurants (27% of revenue)
  Best template: landing-restaurant v7 (+18% vs v5)

Operations
  Avg build time: 3m 12s
  QA fail rate: 4.2% (mostly missing contact form)

Preferences
  Theme: green CTA +12% click-through in previews owner approved

Recommendations (with why)
  → Improve dental template footer (failed QA 9× this month)
  → Propose new “cleaning service” Skill (Ideas score 0.81)
```

Owner sees **reason + confidence**, not black-box magic.

## Memory vs Skills

| | Skill | Company Memory |
|--|-------|----------------|
| What | How to build X | What worked when building many X |
| Changes | Add new capability | Improves proposals and rankings |
| Core impact | None (plug-in) | None (read-mostly service) |

New Skill does not require rewriting Memory. Memory gets better as more lifecycle cycles complete.

## Mature formulation

> **Genesis does not live alone. It evolves as a digital company using accumulated experience and new Skills.**

---

## Build order

1. ◯ Audit events rich enough (task lifecycle) — **partial today**
2. ◯ First Skill lifecycle complete → memory has real product data
3. ◯ Analytics department aggregates stats
4. ◯ Memory Service — summaries for morning brief and Ideas
5. ◯ Evolution proposes template changes from memory — owner approves

**Do not build Memory ML before one Skill ships.**

- Lawful data only (`WHY.md`)
