# Genesis — Roadmap (стратегия развития)

**Статус:** принято · **не переписывать** существующее · реализовывать **постепенно**

**Версия лаунчера:** 0.4.0+ · **Фаза:** R1 — первый продаваемый продукт → первый €

### Правило 80/20 (заморозка идей)

> **80% времени — реализация Roadmap. 20% — новые идеи.**

Roadmap **достаточно полон**. Не добавлять 10–15 концепций подряд. Новый отдел в документ — только если есть **слот в очереди R0–R14** и **условие старта**. Иначе — в backlog одной строкой, без нового кода.

**Сейчас для Cursor:** R1 (цикл продукта). **Vision Freeze** — см. `WHY.md` § Vision Freeze.

---

## Приоритет №1 (навсегда)

> **За каждой кнопкой — реальная работа. Не имитация.**

- Mission Control показывает только то, что произошло в системе.
- Нет данных → **0**, не выдуманные цифры.
- Новый экран или агент — только если есть **источник правды** (событие в логе, запись в БД, завершённая задача).
- См. также `WHY.md` § Gradual evolution, Agent payback rule.

## Блоки Roadmap

### 1. Экран «Развитие Genesis»

**Цель:** видеть, как **реально** растёт интеллект системы — не доход, а способности. Это самый интересный экран Genesis в долгосрочной перспективе.

**Конституция:** `WHY.md` § **Genesis Lifetime Mission** — минимум одно измеримое улучшение в месяц.

**Показывать (только из фактов):**

| Раздел | Источник данных |
|--------|-----------------|
| Новые навыки | CHANGELOG / capability flags после релиза с тестом |
| Ошибки, которые научились чинить | `launcher/logs`, auto-fix successes (счётчик) |
| Новые возможности после обновления | diff capability registry vs предыдущая версия |
| Метрики месяца (когда есть baseline) | build time, error rate, validator score — только сравнение с прошлым месяцем |

**Пример дня (честный):**

```text
Сегодня Genesis умеет (проверено):
✅ Улучшать лендинги после замечаний владельца
✅ Сам исправлять ошибки запуска (npm, .next, порт 3000)
🔄 Публикация — кнопка есть, ждёт первого live-клиента
📅 Следующая цель: Research Agent v1 (одна ниша)
```

**Пример месяца (цель UI, когда накопятся данные):**

```text
За этот месяц Genesis стал лучше:
✓ Освоил новый инструмент для создания 3D-моделей
✓ Научился автоматически оптимизировать Unity-проекты
✓ Улучшил генерацию сайтов
✓ Ускорил сборку проектов на 18%
✓ Снизил количество ошибок на 27%

Следующая цель:
Изучение процедурной генерации уровней
```

Строки с процентами — **только** если есть два реальных замера (до/после). Иначе — не показывать.

**Когда:** после стабильного ежедневного запуска Mission Control.

**Где:** Mission Control → новая секция или `/company/evolution`.

### 2. Уровень автономности

**Цель:** один показатель «насколько Genesis работает сам» — из **реальных** включённых способностей.

**Чеклист (v1):**

| Способность | Сейчас (честно) | Флаг в API |
|-------------|-----------------|------------|
| Создание продукта (Factory) | ✅ | `factory.can_create` |
| Улучшение по feedback | ✅ | `factory.can_improve` |
| Проверка качества (Validator) | ✅ | `validator.active` |
| Публикация | 🔄 частично | `publisher.live` |
| Анализ рынка | ❌ | `research.active` |
| Поиск клиентов | ❌ | `crm.active` |
| Автоматические продажи | ❌ | `sales.autonomous` |

**Формула:** `percent = round(done_count / total_count * 100)` — без ручной подкрутки.

**Сейчас ожидаемо:** ~25–35%, не 60%.

**Когда:** вместе с «Развитие Genesis» или сразу после.

### 3. Живой журнал работы AI

**Цель:** ощущение живой компании — **но только реальные события**.

**Правила:**

- Событие попадает в ленту только если задача **началась** или **завершилась** в backend.
- Запрещено: таймерные фразы «Analyst изучает…» без задачи в очереди.
- Разрешено сейчас: Factory create/improve, Validator result, Publisher approve/publish, launcher auto-fix.

**Формат:**

```text
14:32  Factory     Создан лендинг «Автосервис» (quality 87%)
14:35  Validator   Проверка пройдена
14:40  Publisher   Ожидает одобрения владельца
```

**Техника:** `memory/events.jsonl` или расширение `narrative_feed` — только append из сервисов.

**Когда:** после пункта 1; до Research Agent — только Factory/Validator/Publisher.

### 4. Мозг Genesis (раздел Brain)

**Цель:** отдельный экран накопленного опыта — когда Brain реально пишет данные.

**Метрики (все из storage, не UI):**

- найдено идей (Research Agent — когда появится)
- исправлено ошибок (auto-fix + owner feedback loop)
- принято решений владельцем (approve/reject audit)
- накопленный опыт (products improved, niches tried)
- чему научились (capability changelog)

**Сейчас:** заготовка в `docs/BRAIN_ROADMAP.md` · UI **не строить** до живых событий.

**Когда:** после Research Agent v1 или накопления ≥30 дней owner events.

---

### 5. Все показатели только реальные

Уже в конституции (`WHY.md`). Roadmap закрепляет:

- Доход, клиенты, проекты — из Finance/Factory/CRM, не demo (demo явно помечен).
- «Ночная смена» — не считать исследованием рынка, пока нет Research Agent.
- Value Score / company readiness / autonomy — только вычисляемые поля.

---

### 6. Главная цель — AI-директор компании

**Конечная точка (годы):**

Genesis постепенно берёт на себя:

1. анализ рынка (Research Agent v1 → v2)
2. создание продуктов (Factory — уже v0.1)
3. улучшение (Factory improve + обучение на feedback)
4. поиск клиентов (CRM + Research)
5. помощь в решениях (очередь «требует владельца»)

**Правило развития:** одна способность → тест в реальной работе → измеримая польза → следующая.

**Первый агент после рубежа запуска:** Research Agent v1 (одна ниша, реальный список компаний).

### CEO Mode — следующая глава (после первого честного €)

**Не новый отдел и не новая функция.** Изменение того, **как владелец управляет компанией**.

**Цель:** утром владелец не ищет кнопки — сразу видит:

- сколько заработано (реально);
- какие проекты в работе;
- где нужен ✔;
- что уже делает команда;
- что можно не трогать.

**Принципы CEO Mode:**

- Только реальные данные — никаких демо-цифр.
- Только действия, требующие владельца; остальное — статус отделов.
- Genesis готовит → владелец утверждает.
- Если данных нет — «данных пока нет», не придумывать.

**Пока не реализовывать.** CEO Mode начинается только после:

- ✅ Genesis Stable
- ✅ первого завершённого заказа (R1.5 пройден с реальным клиентом)
- ✅ первого честно заработанного €

До этого — пользоваться Genesis и записывать, какие действия повторяются каждый день.

### Долгосрочное видение — продающий Factory (не сейчас)

**Проблема:** лендинг без реальных контактов, формы заявки и CTA не завершает продажу.

**Цель:** Factory по умолчанию выпускает сайты, **готовые принимать клиентов** — не только красивые.

**Этап 1 (после первого €):** стандарт продающего лендинга:

- сбор данных владельца (телефон, WhatsApp, email, адрес, часы работы);
- форма заявки + кнопки связи;
- карта, доверие, CTA, SEO, mobile;
- валидатор: «без телефона — продукт не готов».

**Этап 2 (после Sales):** AI-консультант на сайте → заявка → уведомление владельцу.

**Этап 3 (долгосрочно):** полный цикл с **явными полномочиями** владельца (ответы ✔, цены ✔, оплата через Payment Hub ✔, возвраты ✔ только с ✔).

Genesis не принимает деньги и не заключает сделки без интеграций и разрешений.

**Сейчас:** Factory v0.1 — placeholder-контакты (`+7 000…`). Исправляется **после** первого тестового прогона R1.5, не параллельно со Stable.

**Architecture Freeze продолжается.** Новые идеи не реализовывать до первого реального дохода. Работать только над тем, что помогает получить следующего клиента и успешно завершить заказ.

### R5 — Opportunity Engine (Движок возможностей)

**Миссия (конституция):** круглосуточно искать **законные** возможности — продукты, клиентов, разрешённые задачи. Выполнять всё, что **не требует** решения владельца. Публикация, договоры, крупные расходы, риск — только после **✔**.

**Не:** «машина для печати денег». **Да:** конвейер возможностей → продукт → клиент → €.

**v1 (первая реализация = бывш. Research Agent):**

| Компонент | v1 scope |
|-----------|----------|
| Источник | одна ниша, один регион — реальные компании (сайт, контакты, объяснимый score) |
| Очередь | `memory/opportunities.jsonl` — append только после скана |
| Ночная смена | 1 запланированный прогон/сутки, не фейковая лента |
| Выход | карточка возможности → «Создать продукт» / «Отклонить» |

**Пример UI (когда v1 live, только факты):**

```text
Найдено сегодня: 12 компаний без современного сайта
Статус: 2 предложения подготовлены · 1 ждёт ✔ владельца
```

**Главный KPI (не количество идей):**

```text
Сегодня найдено возможностей: N   ← реальные записи в очереди
Использовано: M                   ← привели к действию (продукт/письмо)
Получено клиентов: K              ← milestone owner_tested / CRM
Доход: X €                        ← finance, не выдумка
```

**Сейчас:** `engine_active: false`, все KPI = **0**, подпись «Сначала R1 — один продаваемый продукт».

## Genesis Evolution (долгосрочная миссия)

> **Genesis постоянно расширяет законные способности и автоматизирует всё больше полезной работы — постепенно, не магией.**

> **Genesis Lifetime Mission:** каждый месяц — хотя бы одно полезное дело **лучше**, чем месяц назад. Эволюция, не раздувание. См. `WHY.md` § Genesis Lifetime Mission.

Каждая новая способность проходит:

```
Изучение → Прототип → Тест → Проверка владельцем → Реальное применение → Постоянная capability
```

**Железное правило безопасности:**

> Genesis может предлагать, создавать, тестировать и автоматизировать — но **публикация, крупные расходы, рискованные решения и merge в production** только после **✔ владельца**.

## Genesis HQ — штаб всех проектов (фаза R7+, годы)

Genesis — **не одна игра и не один бизнес**. Это **операционная система**, которая **управляет отдельными проектами**, не сливает их код в один репозиторий.

```
GENESIS (штаб)
├── 💼 Genesis Business      ← этот репозиторий
├── 🎮 Perfect Pallet Studio
├── 🌐 Website Factory
├── 📱 Mobile Studio
├── 📣 Marketing & Growth    ← после первого клиента (дизайн)
├── 💰 Finance
├── 🧠 Brain
├── 💼 Opportunity Engine
├── 🎯 Executive AI          ← распределение приоритетов (дизайн)
└── 📊 Investment Engine     ← реинвест прибыли (дизайн)
```

**Архитектурный принцип:** Perfect Pallet **не внутри** Genesis-кода. Genesis хранит **метаданные проекта**, статус, задачи, отчёты, test build — и **пути** к репозиториям на диске.

**Утренний экран (цель — центр управления всеми проектами):**

```text
Доброе утро, Рамиш.

За ночь:
· Factory завершил черновик лендинга
· Perfect Pallet — test build собран

💼 Genesis Business — продукт ждёт решения · 248 €/нед
🎮 Perfect Pallet — Test Build готов · 4 fix · FPS +12% · ✔/❌
🌐 Клиентские сайты — 2 готовы к публикации

Чему Genesis научился за месяц: см. «Развитие Genesis»

Сегодня требуется: одобрить механику · проверить сайт
```

**Агенты по студиям (не смешивать):**

| Business Studio | Perfect Pallet Studio |
|-----------------|----------------------|
| Analyst, Factory, CRM, Publisher, Finance | Game Designer, Unity Dev, QA, Performance, Art, PM |

## Perfect Pallet Studio (после R4–R6)

**Не** непрерывное авто-изменение игры. Цикл:

1. AI готовит изменения в **ветке / sandbox**
2. Собирает **Test Build** → ярлык на рабочем столе
3. Genesis показывает отчёт (ошибки, FPS, changelog)
4. Ты играешь 5–10 минут
5. **✔ Принять** → merge в main · **❌ Отклонить** → откат

Всегда две версии: **рабочая** и **экспериментальная**.

**Метрики только реальные:** % прогресса из задач/милестоунов, не выдуманный «63%».

## Cursor Bridge — «движок разработчика» (после R0–R3)

**Видение:** ты говоришь Genesis → Genesis ставит задачу → Cursor правит код → Genesis тестирует → UI обновляется. Без Alt+Tab.

**Реально, но не сейчас.** Требует:

| Компонент | Назначение |
|-----------|------------|
| Task queue | Задача с project_id, scope, owner_approved |
| Cursor adapter | CLI/API: передать prompt, дождаться, получить diff |
| Watcher | Заметить изменения файлов |
| Rebuild hook | npm/Unity build по типу проекта |
| Verify gate | pytest / play mode / health check перед показом UI |

**Пример потока:**

```
«Сделай лучше раздел Финансы»
  → Genesis → task (project: genesis-business)
  → Cursor → patch
  → verify_release + pytest
  → hot reload / перезапуск frontend
  → Mission Control показывает результат
```

**Ограничения:**

- Авто-merge в production **запрещён** без ✔
- Финансы, платежи, публикация — **всегда** ручное подтверждение
- Один проект за задачу (не «улучши всё»)

**Когда:** R8+ после Capability registry и стабильного daily use. До R0 — **не строить**.

### 📚 Genesis Knowledge (после Brain + живые проекты)

**Не сейчас.** Долгосрочная цель: Genesis **помнит всё проверенное** — не выдумывает, а накапливает audit trail.

**Что хранить (по проекту):**

| Тип | Пример |
|-----|--------|
| Улучшения | «улучшение #201 — паллетная физика» |
| Ошибки и fix | «34 ошибки · 126 решений» |
| Неудачные попытки | «18 отклонено владельцем — причина: FPS −12%» |
| Решения владельца | approve / reject с датой и контекстом |

**Зачем:** чтобы система могла сказать честно:

> «Мы уже пробовали это в январе. FPS упал — предлагаю другой вариант.»

**Правила:**

- Запись только после **реального** события (task completed, test failed, owner rejected).
- Неудачные попытки **так же ценны**, как успехи — иначе нет обучения.
- Привязка к `project_id` (Business vs Perfect Pallet vs client site).
- Источник: `memory/knowledge/` или event log + embeddings (фаза 2).

**Когда:** R11+ после R5 (Brain UI) и ≥90 дней owner events по ≥2 проектам.

**Связь с миссией:** «Я работаю в Genesis» — Knowledge делает Genesis **умнее с годами**, Cursor остаётся инструментом за кулисами.
