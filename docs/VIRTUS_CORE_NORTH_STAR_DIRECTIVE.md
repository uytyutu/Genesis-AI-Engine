# Virtus Core — North Star Directive

## Co-Creation Workspace Vision

### Status

**This document is a product vision, NOT an implementation task.**

Do **not** start implementing this vision automatically.
Do **not** expand current development scope.

Current priority remains:

* Mission 1
* First paying customer
* R4-B user research
* Small validated slices only

This document describes the long-term direction of Virtus Core after validation.

**Stability:** After this version, treat this document as frozen product philosophy. Change the product through validated slices — not by rewriting the North Star.

---

# Core Philosophy

Virtus Core is **not** another AI generator.

Virtus Core is a **Co-Creation Workspace**.

Vector is **not** a chatbot.
Vector is **not** a one-click generator.

Vector is the user's long-term creative partner.

The goal is simple:

> The user should feel that they created the product **together with Vector**, not that an AI generated something behind a black box.

---

# Fundamental Principle

Never force the user to imagine something that can be shown.

Show evolution.
Show drafts.
Show progress.
Show alternatives.

Do not make the user wait in front of an empty loading screen.

---

# Universal Scope

This North Star applies to **all** Virtus Core services — not only websites.

```
Applies to:
• Websites
• Mobile apps
• Desktop applications
• AI systems
• Games
• Business automation
• Marketing projects
• Future Virtus Core services
```

Virtus Core is not a website generator with a chat attached.
It is a Co-Creation Workspace for digital products.

---

# Co-Creation Workflow

Instead of

```
User → Prompt → Wait → Finished result
```

Virtus Core should eventually become

```
User
↓
Conversation
↓
Questions
↓
Live draft
↓
Preview
↓
Discussion
↓
Iteration
↓
Approval
↓
Final build
↓
Delivery
```

The conversation and the creation process happen together.

---

# Co-Creation Commerce

The business lifecycle follows the same philosophy: **price appears after the person has seen the result**, not before.

```
Idea
↓
Conversation
↓
Live Draft
↓
Preview
↓
Approval
↓
Price
↓
Payment
↓
Full Build
↓
Delivery
↓
Long-term evolution
```

The customer pays for confidence — not for a blind promise.
The draft helps the decision. The paid deliverable is the complete production version.

---

# Three Pillars

## 1. Conversation

Vector continuously communicates with the user.

Examples:

* "What feeling should this website create?"
* "Let's choose the visual direction first."
* "I have two possible layouts."
* "Would you like to compare them?"

Conversation replaces long forms and questionnaires.

---

## 2. Visible Progress

The user always knows what Vector is doing.

Examples:

* ✓ Homepage created
* ✓ Navigation drafted
* ⏳ Designing About section
* ⏳ Preparing color palette
* 💡 Idea: Would you like customer testimonials?

No hidden work.
No black box.

---

## 3. Live Preview

The user should see an evolving draft.

Not a finished product after several minutes.
Not only text.

A real visual direction.

Drafts may include:

* page structure
* layout
* navigation
* colors
* wireframes
* application flow
* architecture diagrams
* game level concepts

The goal is orientation, not perfection.

---

# Drafts Before Payment

Before asking for payment, Vector should help the user reach confidence.

The user should be able to understand:

* project direction
* structure
* functionality
* visual identity
* estimated scope

The draft must help the user decide.
The draft must NOT replace the paid deliverable.

---

# Trust-Based Payment

Vector sells through trust — never through pressure.

Not:

```
Pay now.
```

Instead:

```
We have built a concept together.
↓
Are you happy with this direction?
↓
Would you like me to build the complete production version?
↓
Price
↓
Payment
```

Vector should never pressure the user.

Only after the user confirms satisfaction:

1. Present pricing.
2. Ask for confirmation.
3. Proceed to payment.

This is a completely different UX from "prompt → wait → paywall."

---

# Ownership

After successful payment the customer becomes the **owner** of the purchased project.

This is a principle of trust, not a subscription trick.

Depending on project type, ownership may include:

* source code
* project files
* generated assets
* documentation
* configurations
* deployment package
* everything included in the purchased service

Ownership is permanent and independent from future subscriptions.

> **The customer never loses ownership because the subscription expires.**

What was purchased stays with the customer.

---

# Subscription Philosophy

Subscriptions are **not** for locking customer projects.

Not:

```
Subscription unlocks your files.
```

Instead:

```
Subscription allows continuous collaboration with Vector.
```

Subscriptions are for:

* ongoing improvements
* new features
* project evolution
* a long-term creative partner

Examples:

* "Add an online shop."
* "Create an admin dashboard."
* "Translate into German."
* "Improve SEO."

Vector continues working on the existing project.
The customer always works inside the same project.

A lapsed subscription may pause collaboration — it must never revoke ownership of what was already purchased.

---

# Character of Vector

Vector thinks aloud.

Examples:

* "I'm still working on the homepage."
* "I'd like to compare two layouts."
* "I'm waiting before changing Contacts."
* "I recommend finishing the navigation first."

The user should never wonder what Vector is doing.

---

# The Feeling

The customer should never feel that an AI disappeared for five minutes.

The customer should feel that Vector is sitting next to them — thinking, showing, asking, improving, and building together.

The desired future user sentence is NOT

> "The AI created my website."

The desired sentence is

> **"We built this website together."**

That is the emotional North Star.

---

# Scope Protection

This vision must NOT automatically create implementation work.

Future implementation begins ONLY when validated by user research.
R4-B remains the gate.

Only after repeated real-world evidence should implementation begin.

Even then:

* Implement one brick at a time.

Possible first bricks:

* live draft panel
* activity timeline
* project evolution preview
* draft approval workflow

Never implement the complete workspace in one cycle.

```
Small slice → validation → commit → soak → next slice
```
